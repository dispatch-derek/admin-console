# F-002: Set a Customer-Specific System Prompt

*A console-managed, customer-wide baseline system prompt applied across all of a customer's workspaces.*

## Problem
There is no way, in the admin console, to establish a consistent baseline system prompt — a persona, tone, and safety/guardrail foundation — across all of a customer's workspaces. Each workspace's system prompt is configured independently (a single free-text field per workspace), so the intended baseline can drift between workspaces or be omitted entirely when a workspace is set up, with no single source of truth guaranteeing every workspace carries it. Staff operators configuring a customer's instance have no mechanism to define "this is the baseline every workspace must include," so consistency depends on manual discipline across an unbounded number of workspaces.

*(Note: evidence is the structural fact that prompts are per-workspace with no baseline mechanism, plus conviction about the guardrail need — no support-ticket or analytics signal was gathered, since no discovery scan was run. Honest thinness, not a hidden gap.)*

## Affected Users
The customer's end-users experience the system prompt in every chat, in every workspace — so the baseline (or its absence) shapes essentially all of the customer's chat interactions. The people who would configure the baseline are our internal staff operators (workspace settings are staff-console-only; customers do not edit workspace prompts). Configuration is an infrequent, high-leverage action; its effect is continuous and pervasive across all workspace chats for that customer.

## Business Rationale
Stated as falsifiable claims:
- **Brand/safety consistency (risk reduction):** a guaranteed baseline persona/guardrail across all workspaces reduces the chance of off-brand, off-tone, or unsafe responses in any single workspace whose prompt was misconfigured or left bare. Falsifiable by auditing prompt consistency across workspaces before/after, and by tracking guardrail-omission incidents.
- **Operator effort:** setting one baseline instead of re-authoring the same foundation per workspace lowers configuration effort and drift-maintenance as workspace count grows. Falsifiable by measuring per-customer prompt-setup time and re-sync effort.
- **Governance as a selling point:** a customer-level prompt-governance capability is a plausible differentiator for customers who need centralized control. Falsifiable by whether it appears in sales asks / closes (currently unverified — see Open Questions).

## Timing
Tied to the **October 2026 go-to-market** (~3 months out from 2026-07-07). Baseline-prompt governance is plausibly part of the GTM configuration/onboarding story. The cost compounds: every customer and workspace onboarded before this exists accrues manually-set prompts that must be retrofitted to a baseline later, and the retrofit surface grows with the workspace count.

## Existing Evidence
Pointers only — leads to re-verify at scoring time, not established fact:
- **Existing per-workspace mechanism (internal):** `web/src/features/workspaces/WorkspaceSettings.tsx` (system-prompt textarea, "blank = inherit"), `workspace.systemPrompt` → engine `openAiPrompt` (`bff/src/engine/mappers.ts`, `engine-types.ts`). This is the API-key-reachable channel the console-managed baseline would write through.
- **Feasibility boundary (internal):** `docs/anythingllm-surface.md` — the native instance-level **Default System Prompt** (`/system/default-system-prompt`) is session-auth-only and **not reachable via the console's API-key custody model** ("a hard boundary at the API layer"). This is *why* F-002 takes the console-managed route rather than driving the native default. Also relevant: `/system/prompt-variables` (native, out of scope).
- **Console patterns available:** existing WorkspaceList + per-workspace PATCH, the `DangerConfirm` typed-token flow, `ErrorBanner`, and the REQ-098 no-partial-success handling already in the settings slice.
- **Not gathered:** no support-ticket, customer-request, or analytics signal — no discovery scan was run for this brief.

## Proposed Direction
*(Non-binding sketch.)* Add a console-level "customer baseline system prompt" that staff define once, which the console applies across all of the customer's workspaces through the existing per-workspace system-prompt API (the native instance-level Default System Prompt is deliberately avoided — not reachable under the console's API-key custody model). The console would persist the baseline itself (new console-side data) and write it into each workspace's prompt. How the baseline composes with any per-workspace prompt (overwrite, prepend-baseline-then-workspace-specific, or fill-only-when-blank), how it re-syncs when the baseline changes, and how newly created workspaces inherit it are left open for design. Enforcement is best-effort within the engine's single-prompt-field model — the engine has no native "base + workspace" layering, so a true tamper-proof guardrail is not achievable through this channel alone.

## Design Considerations
*The following reads are from the ux-designer agent (Phase 3b); they inform — not set — the human's later Effort and Risk scores:*

- **complexity_read:** A new console-level settings surface introduces the app's first "above-workspaces" scope — today every settings form is bound to a single workspace, so this needs a home (a new settings section/route) that has no sibling in the current navigation. The baseline field itself maps cleanly onto the existing data-driven form idiom (a labeled textarea like the current "System prompt (blank = inherit)"), so the input control is familiar and low-novelty. The genuinely new interaction surface is everything around the write, not the field: the composition-mode choice left open in the brief (overwrite vs prepend vs fill-when-blank) is a semantic control with no precedent here, and each option implies different supporting UI — a preview of the composed prompt (baseline + per-workspace) is a pattern the console has never needed, since per-workspace prompts are currently just opaque free text. A bulk-apply/sync affordance that writes across an unbounded set of workspaces is also novel: the app has per-entity saves and a single-entity DangerConfirm token flow, but nothing that fans one action out to many targets or reports partial success across them. Related novel surfaces the brief invites: per-workspace override/drift visibility (showing which workspaces diverge from the baseline) and how newly created workspaces surface their inheritance — both would extend the existing WorkspaceList/WorkspaceSettings views with status the data model doesn't currently express. Design-system fit is otherwise good: ErrorBanner, the list/detail column layout, blank=inherit conventions, and the DangerConfirm typed-token pattern are all reusable building blocks for these new pieces.

- **ux_risk_read:** The defining risk is blast radius plus reversibility: a single baseline save can rewrite the system prompt of every workspace, and because the engine has one prompt field, an overwrite or careless compose mode can clobber operator-authored per-workspace prompts with no native undo — the console would be the only place any prior value survives, if at all. That raises a strong need for pre-write comprehension: staff should be able to see what will change, on which workspaces, before committing (a diff/preview and an affected-count), and the write should degrade visibly on partial failure since best-effort fan-out means some workspaces may end up synced and others not, leaving an inconsistent state the UI must make legible rather than hide. Drift/override visibility is both a feature and a risk surface — if staff can't tell which workspaces still carry the baseline versus have diverged, the "single source of truth" promise silently erodes and the tool gives false confidence. This is high-leverage, infrequent, and hard to eyeball for correctness, so it is a strong candidate for task-based usability observation of a staff operator performing a first-time apply and a later re-sync, watching for whether they correctly predict the effect on existing per-workspace prompts. Reversibility mitigations worth designing for (confirmation gating on the destructive compose modes, echoing the existing typed-token danger pattern; a dry-run/preview step) reduce but do not remove the irreversibility once written to the engine. Accessibility exposure of the new surface is moderate and mostly inherited: the field and form follow existing label/aria-invalid patterns, but the new elements — a composed-prompt preview, a multi-workspace affected-list, and any confirm dialog — need deliberate focus management, status messaging announced to assistive tech (especially async bulk progress and partial-failure results), and non-color-only encoding of drift/synced status.

## Out of Scope
- The native AnythingLLM instance **Default System Prompt** (`/system/default-system-prompt`) and **Prompt Variables** (`/system/prompt-variables`) — session-auth-only, outside the console's API-key custody model.
- Per-workspace system-prompt *editing* itself — already exists, and workspace-level prompting is F-003's territory.
- Any change to the customer-facing chat UI or end-user experience beyond the prompt content it already consumes.
- True tamper-proof enforcement (not achievable via the engine's single prompt field alone).

## Open Questions
- **Composition semantics (the crux):** overwrite / prepend-baseline-then-workspace-specific / fill-only-when-blank / other? How is per-workspace customization preserved while guaranteeing the baseline "isn't silently lost," given the engine's single prompt field has no native layering?
- **Enforcement strength:** one-time apply, or continuously enforced (re-sync on drift, auto-apply to new workspaces)? May staff still override per workspace, and is that permitted?
- **Persistence:** where does the console store the baseline (BFF-side data model) and how does it track per-workspace sync/drift state?
- **Baseline-change propagation:** on edit, bulk re-push to all workspaces? Partial-failure handling (some PATCHes fail)?
- **Conflict with native default:** a customer could set AnythingLLM's native default prompt out-of-band (console can't read/write it) — how do the two interact?
- **Scale:** typical workspaces-per-customer (bounds the bulk-apply effort and the drift-visibility UI)?
- **Guardrail tamper-resistance:** if "guardrail" implies the baseline must survive workspace edits, single-field storage can't truly enforce it — is best-effort acceptable, or is this a hard requirement (which would change the whole approach)?
