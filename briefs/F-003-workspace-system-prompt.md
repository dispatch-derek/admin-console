# F-003: Set a Workspace-Level System Prompt

*The workspace prompt as a first-class, composable layer over the customer-wide baseline (F-002) — not a lone opaque field.*

## Problem
The console already exposes a per-workspace system prompt, but only as a single opaque free-text field (`web/src/features/workspaces/WorkspaceSettings.tsx` — "System prompt (blank = inherit)"). That field has no defined relationship to anything above it. Once F-002 introduces a customer-wide *baseline* system prompt applied across all of a customer's workspaces, a staff operator configuring one workspace cannot answer the question that actually matters: *what prompt effectively governs this workspace's chats?* There is no way to express "this workspace adds its own instructions on top of the baseline" versus "this workspace overrides the baseline," no view of the effective composed prompt, and no reconciliation between the workspace field's existing "blank = inherit" convention and the new baseline layer. The workspace prompt and the customer baseline both write the engine's single prompt field with no model for how they combine, so the intended layering is invisible and operator-unpredictable.

*(Note: evidence is the structural fact that the workspace prompt is a lone field with no composition model, plus conviction that the layering gap becomes real once F-002 lands — no support-ticket or analytics signal was gathered; no discovery scan was run. Honest thinness, not a hidden gap.)*

## Affected Users
Internal staff operators configure workspace prompts (workspace settings are staff-console-only; customers do not edit them). The customer's end-users experience the *effective* prompt in every chat in that workspace, so the composition outcome (or its ambiguity) shapes all of that workspace's chat interactions. Configuration is infrequent and high-leverage; its effect is continuous and pervasive across every chat in the workspace. Reach is bounded by a small operator base (F-001 ruling OQ-5: <25 staff) but each configuration action governs an unbounded stream of downstream chats.

## Business Rationale
Stated as falsifiable claims:
- **Governance completeness (pairs with F-002):** F-002's customer baseline is only half a prompt-governance story; without a defined workspace layer, operators either can't customize per workspace without silently clobbering the baseline, or can't guarantee the baseline survives workspace edits. Falsifiable by whether operators can both customize a workspace *and* retain the baseline after F-003, versus not before.
- **Operator predictability / error reduction:** an effective-prompt view plus explicit compose modes reduces misconfiguration of a high-leverage, hard-to-eyeball field. Falsifiable by tracking prompt-misconfiguration incidents and operator ability to predict the governing prompt in a task check.
- **Configuration velocity:** reusing a baseline and layering only workspace-specific deltas lowers per-workspace authoring effort versus re-stating shared instructions in every workspace. Falsifiable by measuring per-workspace prompt-setup time.

## Timing
Tied to the **October 2026 go-to-market** (~3 months out from 2026-07-07). F-003 is the workspace half of the layered-prompt capability F-002 opens; the two are most valuable delivered together for the GTM configuration/onboarding story. The cost compounds with F-002's: every workspace prompt authored before a composition model exists accrues as an opaque field whose relationship to the later baseline must be retrofitted, and that retrofit surface grows with workspace count.

- **Feasibility boundary (internal):** the engine exposes exactly one prompt field per workspace (`openAiPrompt`); any "baseline + workspace" layering is computed console-side and flattened into that one field — the engine has no native prompt composition (`docs/anythingllm-surface.md`).

## Proposed Direction
*(Non-binding sketch.)* Promote the workspace-level system prompt from a standalone textarea to a first-class layer relative to the F-002 customer baseline. Let the operator choose, per workspace, how the workspace prompt combines with the baseline (e.g. append-to-baseline / override-baseline / inherit-baseline-only), and surface the **effective composed prompt** the workspace will actually run with. Because the engine holds a single `openAiPrompt` field, composition is computed console-side and written into that one field; the console would need to track the workspace-specific "layer" separately from the flattened result so the baseline and workspace contributions remain distinguishable across edits and baseline changes. How this reconciles with F-002's baseline drift/re-sync model, what "blank = inherit" means in a layered world, and whether an override path is destructive enough to gate are left open for design.

## Design Considerations
*The following reads are from the ux-designer agent (Phase 3b); they inform — not set — the human's later Effort and Risk scores:*

- **complexity_read:** No net-new top-level screen is implied. The direction extends the existing workspace detail editor (`web/src/features/workspaces/WorkspaceSettings.tsx`) — a data-driven settings form the console already has — rather than adding a route. What it adds is a new *sub-region* inside that form, replacing today's lone opaque "System prompt (blank = inherit)" textarea with a small cluster: a per-workspace composition-relationship control, the workspace-layer text field, and a derived "effective composed prompt" view. Novel interaction patterns relative to what the workspace editor does today: (1) a per-workspace mode/relationship selector (append-to-baseline / override / inherit-only) — conceptually adjacent to F-002's per-apply mode selector, but here per-workspace and persistent, not a one-shot operation choice; (2) a read-only, live-recomputing "effective prompt" preview embedded in an editable form — the console has a preview/diff concept in F-002 but at the customer-baseline surface, not inline in per-workspace editing; (3) reinterpreting the established "blank = inherit" idiom (today blank means "inherit the engine default"; in a layered world it must mean "inherit the baseline layer only" — same phrase, changed meaning across the form's convention). Design-system fit is largely favorable for the visible widgets: the mode selector maps onto existing Select/radio conventions (post-F-001 DS `Select`/`Toggle`); the workspace-layer field reuses the existing `systemPrompt` textarea idiom; validation reuses the `aria-invalid`/field-error patterns and partial-write semantics; `ErrorBanner` and REQ-098 handling are reusable as-is; `DangerConfirm` exists if an override path needs gating. The complexity does not concentrate in screen count or in finding components — it concentrates in **cross-feature coupling**: F-003 and F-002 both write the *same single* engine `openAiPrompt` field, so F-003's composition must reconcile with F-002's SENTINEL/remainder/drift model. The proposed direction explicitly leaves that interplay open, and F-002 may not yet be built — so the hardest design integration is agreeing one composition/effective-prompt model across two features rather than laying out any one screen.

- **ux_risk_read:** The host form already carries the console's label/`aria-invalid` conventions, so the accessibility baseline is inherited; the new deltas are what to watch. The mode/relationship control must be keyboard-operable and must not signal meaning by color alone. The "effective composed prompt" preview is a live-updating read-only region: it needs correct semantics so assistive tech does not treat it as editable, and a deliberate decision on whether recomputation is announced (a live region risks being chatty as the operator types; silence risks the change going unnoticed by non-sighted operators). Reversibility: engine prompt writes have no native undo and the engine holds exactly one prompt field, so a save that composes baseline + workspace layer overwrites whatever was there. The pre-save interaction is reversible (change mode/text and re-inspect the effective preview before submitting), but the save itself is not, and a prior hand-authored per-workspace prompt can be lost if composition mishandles the workspace "remainder." As sketched, F-003 has not defined its own remainder bookkeeping the way F-002 has; without that, the low-reversibility of the write lands on a pattern whose prior-state preservation is undefined — that is where the irreversibility risk is real. The preview is the primary mitigation, which raises the bar on it being trustworthy and accurate. Usability-testing needs: the central comprehension risk is the layering mental model itself — an operator must correctly predict "what prompt actually governs this workspace's chats" from baseline + workspace-layer + chosen relationship. Whether operators read the preview, trust it, and correctly map the mode names to outcomes is exactly what a lightweight comprehension check would de-risk; the reinterpreted "blank = inherit" is a known confusion vector worth probing specifically. Reach is bounded (small operator base, infrequent action), but each action is high-leverage and its effect is continuous across every chat in the workspace, so a single misunderstanding propagates silently and broadly — the cost-per-error is high even though the population is small, which argues for validating the mental model before build rather than after.

## Out of Scope
- The **customer-wide baseline** itself (defining, storing, and applying it) — that is F-002. F-003 consumes the baseline as the layer it composes against; it does not create it.
- The native AnythingLLM instance **Default System Prompt** and **Prompt Variables** — session-auth-only, outside the console's API-key custody model (see F-002 out-of-scope).
- Any change to the customer-facing chat UI or end-user experience beyond the prompt content it already consumes.
- Prompt *templating/variables/versioning* as a management surface — a richer capability that could be a separate feature; F-003 is scoped to the composition layer, not a template system.
- True tamper-proof enforcement of either layer (not achievable via the engine's single prompt field alone).

## Open Questions
- **Composition model (the crux, shared with F-002):** which relationships are supported (append / override / inherit-only / other), and how does F-003's per-workspace composition reconcile with F-002's baseline drift/remainder/sentinel model, given both write one engine field?
- **Remainder bookkeeping:** does F-003 introduce its own tracking of the workspace-specific layer (so baseline vs workspace contributions stay distinguishable across edits), or defer entirely to F-002's mechanism? What happens to a workspace's layer when the baseline changes?
- **"Blank = inherit" semantics:** what does an empty workspace prompt mean in the layered world — inherit baseline only? And how is that communicated so it doesn't read as "inherit engine default" (its current meaning)?
- **Override destructiveness:** is switching to override (which can discard baseline governance for that workspace) a `DangerConfirm`-gated action?
- **Effective-prompt preview fidelity:** is the previewed composition guaranteed to equal what the engine runs, and is it recomputed live as the operator edits?
- **Sequencing with F-002:** can F-003 ship before/independently of F-002, or is a built baseline a hard prerequisite?
- **Scale:** typical workspaces-per-customer (bounds the drift-visibility and re-sync surface where a baseline change ripples into workspace layers)?
